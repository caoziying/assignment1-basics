### 1. 排序问题
* **表象：** 合并出来的 Token 和参考答案一模一样，但先后顺序出现问题。
* **根本原因：** 当两个 Pair 出现频次相同时（平局），原代码使用的是 `max(pair_count, key=lambda k: (pair_count[k], k))`。这里的 `k` 是 **Token ID 的元组**（如 `(256, 108)`）。这导致算法错误地认为“越晚合并出来的新 Token（ID 越大）优先级越高”，而实际上我们应该使用的是 **原始的 Token ID**。
* **解决方案：** 把 ID 映射回真实的底层的**字节串 (bytes)**，并按照真实的**字典序**来打破平局。
使用 `max(pair_count, key=lambda k: (pair_count[k], vocab[k[0]], vocab[k[1]]))`：
* `pair_count[k]`：找出频次最高的。
* `vocab[...], vocab[...]`：在频次相同时，严格按照 **字节的 ASCII 字典序** 从小到大挑选。

### 2. 带特殊 token 的处理
* **表象：** 合并出来的 Token 出现问题。
* **根本原因：** 
- `for line in f:` 这种经典的按行读取方式，在底层其实是以换行符 `\n` 为界把文件切碎了。这意味着无论原文中有多少个连续的空行，每次交给正则引擎的 `line` 字符串的末尾**最多只有一个`\n**`。正则引擎根本看不见完整的 `\n\n`，自然也就统计不到它。
- 用一个正则表达式进行所有的匹配（包括特殊Token）

* **解决方案：** 
- (在windows上) 用raw.replace(b"\r\n", b"\n").replace(b"\r", b"\n").decode("utf-8")处理读取后的 \r\n
- 放弃按行读取。对于 5MB 这种级别的语料，直接使用 `f.read()` 将整个文件作为一个巨大的字符串加载到内存中，让正则引擎一次性扫过所有的换行符。
- 先找特殊 Token，然后处理其前面位置到上一个 特殊Token的后一个位置的一般Token，最后一段分开处理。

### 3. 算法复杂度爆炸（速度太慢 / 超时）
* **表象：** 测试要求 1.5 秒内跑完，而代码花了 2-3 秒甚至更久，直接触发 Timeout 报错。
* **根本原因：** 在最初的逻辑中，每一轮合并（哪怕只合并一个极其生僻的 Pair），代码都会**把整个字典里所有单词的所有字符对，从头到尾重新统计一遍** (`tc = Counter()`)。随着词表变大、合并次数增多，这变成了极其可怕的嵌套循环运算（时间复杂度飙升）。
* **解决方案：** * **全局只统计一次：** 在 `while` 循环外，先把所有单词的初始配对频次统计好。
* **精准局部修改：** 当我们要把 `A` 和 `B` 合并成 `AB` 时，先减去原字符串的所有匹配对，然后进行合并，在加上合并字符串的匹配对，最后删掉 AB 的cnt。
